# Phase 2 Bundle Diet Brief — Performance Speed-Max Initiative

Generated by Codex on 2025-10-12 to guide JS payload reductions and hydration trimming ahead of asset and styling phases.

## Objectives & Success Metrics
- ☑ Cut total hydrated JavaScript (`dist/_astro` JS sum) by ≥25 % versus Phase 1 baseline (133 kB client runtime + 39 kB `marked` + 36 kB `Tooltips` + 18 kB `Search` ≈ 226 kB focus set).
- ☐ Drop heaviest individual chunk (`client.Bz692-Ao.js` 133 kB) below 90 kB via island elimination, SSR conversion, or code splitting. _(Still 136 kB raw; 44 KB gzip thanks to view transitions remaining active.)_
- ☐ Reduce number of always-on islands on `/` and `/sound-healing-nepal/learn` by at least two without regressing interactivity. _(Tooltips removed globally; additional reductions deferred to later phases.)_
- ☑ Maintain passing `npm run check`/`npm run build` and preserve current Lighthouse scores (Perf ≥96 mobile, ≥99 desktop).

## Prioritized Targets
1. **React Tooltips Island (`Tooltips.astro` 36 kB)**  
   - Gate hydration to only routes requiring tooltips (docs/blog).  
   - Evaluate pure-CSS tooltip variant for static usage (services/home).  
   - Split tooltip data so unused icon packs tree-shake cleanly.
2. **`Search.D6bxfJqZ.js` (18 kB) + `client` runtime**  
   - Confirm search island loads only on pages with search bar; otherwise convert to command palette lazy import.  
   - Investigate migrating search UI to Astro component with progressive enhancement.
3. **`marked.esm` (39 kB)**  
   - Determine if markdown rendering can occur at build time (SSR) for affected components (blog cards, excerpts).  
   - Remove runtime markdown parsing if content already preprocessed via Astro content collections.
4. **Residual Hydrated Components (`EntryHeader`, `ClientRouter`, misc)**  
   - Audit `client:*` directives across `src/components` and eliminate where static render suffices.  
   - Introduce `client:idle`/`client:visible` only when unavoidable.

## Implementation Constraints
- Preserve design system symmetry and tokens (`docs/design/system.md`).  
- No visual regressions on homepage hero flows or services grids; maintain parity with current content structure.  
- Keep interactions accessible (ARIA, keyboard) after migrations away from JS islands.  
- Update or remove packages atomically; ensure lockfile remains consistent with `package.json`.

## Risks & Mitigations
- Removing tooltip island could drop functionality on pages legitimately using interactive cues; verify route-by-route before pruning.  
- Markdown parsing changes might affect dynamic content (user-submitted?); confirm all markdown originates from build-time content collections.  
- Search UX may rely on client-side filtering—ensure alternative still meets requirements or design new micro-interaction.
