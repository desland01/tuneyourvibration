# Phase 3 Asset Optimization Brief — Performance Speed-Max Initiative

Generated by Codex on 2025-10-12 to guide responsive imagery, font hygiene, and media delivery refinements before CSS/script tuning.

## Objectives & Success Metrics
- ☐ Improve homepage and `/sound-healing-nepal/learn` LCP by ≥150 ms via responsive imagery and background compression while preserving visual fidelity. _(Mobile Lighthouse still reports ~3.9 s LCP on the hero H1; investigate fonts/prefetch in next pass.)_
- ☑ Ensure all hero/service imagery ships AVIF/WebP (with fallback) and includes `sizes` metadata so the browser never downloads >1.5× the rendered resolution.
- ☑ Reduce large static background assets (currently 1.7–2.6 MB PNGs under `src/assets/backgrounds`) to ≤400 kB optimized variants without banding. _(Desktop/mobile backgrounds now served via 103 kB AVIF + 88 kB JPG fallbacks.)_
- ☑ Verify the font preload stack stays minimal: Inter (400/700) + Cormorant (700) only, with render-blocking preloads limited to weights actually used above the fold. _(Added Cormorant 500 preload to match hero usage.)_
- ☐ Maintain Lighthouse scores ≥96 (mobile) / ≥99 (desktop) after changes. _(Desktop passes at 100; mobile slipped to 87.)_

## Current Findings
- Hero components already use `astro:assets` `<Image>` but lack `sizes`, forcing 960 px downloads on mobile. Service entries follow the same pattern.
- Global background uses `/images/nepal-day.webp` and `/images/nepal-night.webp` for desktop, but fallbacks exist as heavier PNGs (`bwca-*.png`) in `src/assets/backgrounds/`.
- Search and docs pages include multiple inline images sourced from `src/assets/website-block` and `public/images/uploads` without route-specific compression or format consistency.
- Fonts: `public/fonts/` only contains Inter Regular/Bold and Cormorant 500/700. `BaseLayout` preloads Inter Regular/Bold + Cormorant 700. Cormorant 500 is declared in CSS but not preloaded; confirm whether it’s heavily used above the fold.

## Implementation Plan
1. **Responsive Imagery**
   - Add `sizes` attributes to `<Image>` calls in hero/service components (`home/EntryLayout.astro`, `services/EntryLayout.astro`, `sound-healing-nepal/learn.astro`) reflecting layout breakpoints.
   - Re-export hero assets from `src/assets/website-block` at multiple widths via `Image` (Astro) or `getImage` helpers to ensure `srcset` covers 320→1440 px.
   - Audit markdown/collection imagery (recipes/blog cards) for static file imports; ensure each uses `Image` with constrained width and lazy loading.

2. **Background Assets**
   - Convert `src/assets/backgrounds/bwca-*.png` to AVIF/WebP variants and update references in `Background.astro`; keep PNG fallback only if needed for Safari <16.4.
   - Assess `public/images/nepal-night.jpg` (duplicate of WebP) and remove or ensure it’s only used as fallback.
   - Document resulting file sizes (before/after) in artifacts folder.

3. **Font Stack Review**
   - Confirm where Cormorant 500 appears (e.g., headings/body). If used above the fold, consider preloading; if not, explore dropping the weight to save bytes.
   - Ensure `fonts.scss` retains `font-display: swap` and that no additional weights are imported via third-party modules.

4. **Pipeline Adjustments**
   - For any assets living in `public/images/uploads`, migrate frequently-used items into `src/assets` so Astro pipelines them (enabling automatic format negotiation).
   - If certain hero assets stay in `public/`, add manual AVIF/WebP variants and update markup accordingly.

5. **Validation**
   - Re-run `npm run build -- --verbose` and capture new asset stats (focus on `_astro` image outputs).
   - Serve build via `npx serve` and collect Lighthouse (mobile/desktop) for `/` and `/sound-healing-nepal/learn`.
   - Store raw reports under `docs/research/artifacts/2025-10-12-phase-3/`.

## Risks & Watchpoints
- Over-aggressive compression could introduce banding in gradient-heavy imagery; test on retina screens.
- Removing PNG fallbacks may impact legacy browsers—validate target audience or keep a single lightweight fallback.
- Adding more `<Image>` imports to markdown content may require regenerating MDX frontmatter or adjusting `astro:autoImport` entries.

## Follow-up Questions
- Should homepage background still display imagery on desktop, or can we lean on gradient-only for the fastest path? (Strategic brand decision.)
- Are there upcoming galleries/portfolios that need unified image handling? If so, build shared utilities now.
- Any forthcoming video/animation assets that warrant preloading or lazy loading hooks?
